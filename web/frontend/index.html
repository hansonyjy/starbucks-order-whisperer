How it works

Under the hood it's a 3-stage retrieval pipeline — the same architecture that won the UCLA × Starbucks Data Challenge. Here's what happens between your query and the ranked list.

1

Extract — natural language → structured constraints

Your query is sent to Claude (Haiku 4.5) with a strict JSON schema attached. The model is forced by structured output to return valid JSON — no regex, no markdown-stripping, no parse failures. It reads things like "no dairy", "pick me up", or "under $5" and maps them to typed fields: dairy_free: true, caffeine_level: "high", max_price: 5. A subtle but important rule: ambient weather ("it's hot out") does not set temperature: iced.

2

Filter — hard rules knock out non-qualifying drinks

Pure deterministic logic. Category, temperature, calorie cap, sugar cap, price cap, dairy-free, vegan, and caffeine level all become strict boolean filters. Caffeine "levels" map to mg ranges derived from the training data — including a deliberate medium/high overlap at 150–200mg, because that's what the ground-truth labels say. If the filters are too strict and produce zero results, constraints are relaxed progressively (price first, then sugar, then calories) until something survives.

3

Rank — constraint margin is primary, semantic is tiebreaker

Each surviving drink gets a constraint margin score: how comfortably it sits inside the limits. Under a 250-cal cap, a 0-cal drink scores higher than a 200-cal one. Caffeine-seeking queries rank by most caffeine first; "keep it mild" queries rank by least. Semantic similarity (a local all-MiniLM-L6-v2 embedding) breaks ties between drinks the constraints can't separate — but it's bounded by a tiny ε = 0.001 so it can never flip two drinks whose constraint scores genuinely differ. No paid embedding API, no rate limits, runs offline.
